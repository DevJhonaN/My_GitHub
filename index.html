Git e GitHub - Guia Completo para Iniciantes

O que é Git?

Git é um sistema de controle de versão distribuído, criado por Linus Torvalds em 2005. Ele permite que você acompanhe as mudanças no seu código ao longo do tempo, trabalhe em equipe de forma organizada e mantenha um histórico completo do desenvolvimento do seu projeto.

Imagine o Git como uma máquina do tempo para o seu código. Você pode voltar para qualquer versão anterior, ver exatamente o que mudou e quando, e trabalhar em diferentes funcionalidades simultaneamente sem bagunçar o código principal.

O que é GitHub?

GitHub é uma plataforma online que hospeda repositórios Git na nuvem. É como um "Google Drive" para desenvolvedores, mas muito mais poderoso. Além de armazenar seu código, o GitHub oferece ferramentas para colaboração, controle de issues, wikis e muito mais.

Enquanto Git é a ferramenta que você usa localmente no seu computador, GitHub é o serviço online onde você pode compartilhar seus projetos com o mundo e colaborar com outros desenvolvedores.

Conceitos Fundamentais do Git

Repository (Repositório)

Um repositório é como uma pasta do seu projeto, mas com superpoderes. Ele contém todos os arquivos do projeto e todo o histórico de mudanças. Pode existir localmente no seu computador e/ou remotamente no GitHub.

Commit

Um commit é como um "checkpoint" ou "save" do seu projeto. Cada commit representa um estado específico dos seus arquivos em um determinado momento. É acompanhado de uma mensagem descrevendo o que foi alterado.

Branch (Ramificação)

Uma branch é como uma linha paralela de desenvolvimento. A branch principal geralmente se chama "main" ou "master". Você pode criar outras branches para desenvolver novas funcionalidades sem afetar o código principal.

Merge

Merge é o processo de combinar mudanças de uma branch com outra. Por exemplo, quando você termina uma funcionalidade em uma branch separada, você faz merge dela com a branch principal.

Working Directory, Staging Area e Repository

Working Directory é onde você trabalha nos arquivos. Staging Area é onde você prepara os arquivos que serão commitados. Repository é onde ficam salvos todos os commits.

Fluxo Básico do Git

Passo a Passo do Fluxo Git:

1. Inicializar ou Clonar: Você começa criando um novo repositório ou clonando um existente.

2. Trabalhar nos Arquivos: Você edita, cria ou exclui arquivos no seu Working Directory.

3. Adicionar ao Staging: Você seleciona quais mudanças quer incluir no próximo commit usando git add.

4. Fazer Commit: Você salva as mudanças do Staging Area no repositório com git commit.

5. Enviar para Remoto: Você envia seus commits para o GitHub usando git push.

Comandos Básicos

git init - Inicializa um novo repositório
git clone [url] - Clona um repositório existente
git add [arquivo] - Adiciona arquivo ao staging
git add . - Adiciona todos os arquivos modificados
git commit -m "mensagem" - Faz commit com mensagem
git push - Envia commits para o repositório remoto
git pull - Puxa mudanças do repositório remoto
git status - Mostra o status dos arquivos
git log - Mostra histórico de commits

Fluxo do GitHub

Repositório Local vs Remoto

O repositório local fica no seu computador, onde você trabalha nos arquivos. O repositório remoto fica no GitHub, onde você compartilha seu código e colabora com outros.

Push e Pull

Push: Envia suas mudanças locais para o GitHub. É como "subir" seu código.

Pull: Traz mudanças do GitHub para o seu repositório local. É como "baixar" atualizações.

Fork e Pull Request

Fork: Cria uma cópia de um projeto de outra pessoa na sua conta do GitHub.

Pull Request: Solicita que suas mudanças sejam incorporadas ao projeto original. É como pedir: "Olha as melhorias que fiz, você pode incluir no projeto?"

Trabalhando com Branches

Branches são fundamentais para um fluxo de trabalho organizado. Elas permitem que você trabalhe em diferentes funcionalidades simultaneamente sem bagunçar o código principal.

Fluxo com Branches:

1. Criar uma nova branch para sua funcionalidade

2. Trabalhar nessa branch, fazendo commits normalmente

3. Quando terminar, fazer merge com a branch principal

4. Deletar a branch da funcionalidade (opcional)

git branch [nome] - Cria uma nova branch
git checkout [nome] - Muda para a branch
git checkout -b [nome] - Cria e muda para nova branch
git merge [nome] - Faz merge de uma branch
git branch -d [nome] - Deleta uma branch

Como Resolver Conflitos

Conflitos acontecem quando duas pessoas modificam a mesma linha de código em branches diferentes, ou quando você tenta fazer merge de mudanças incompatíveis. Não se preocupe, é normal e tem solução!

Quando Conflitos Acontecem:

  • Duas pessoas editam a mesma linha do mesmo arquivo
  • Uma pessoa deleta um arquivo que outra pessoa modificou
  • Mudanças simultâneas em arquivos relacionados

Como Identificar um Conflito:

Quando há conflito, o Git para o processo de merge e mostra uma mensagem como:

"CONFLICT (content): Merge conflict in arquivo.txt"

O Git também marca os conflitos diretamente no arquivo com símbolos especiais.

Marcadores de Conflito:

<<<<<<< HEAD
Seu código (branch atual)
=======
Código da outra branch
>>>>>>> nome-da-branch

Passo a Passo para Resolver Conflitos:

1. Identificar o Conflito: Use git status para ver quais arquivos têm conflitos.

2. Abrir o Arquivo: Abra o arquivo em conflito no seu editor de texto.

3. Escolher a Versão: Decida qual código manter:

  • Manter apenas o seu código (remover o código da outra branch)
  • Manter apenas o código da outra branch (remover o seu)
  • Combinar ambos os códigos de forma inteligente
  • Reescrever completamente a seção

4. Limpar os Marcadores: Remova todas as linhas com <<<<<<<, =======, e >>>>>>>

5. Testar o Código: Certifique-se de que o código funciona após a resolução.

6. Adicionar e Commitar: Use git add e git commit para finalizar o merge.

Comandos Úteis para Conflitos:

git status - Ver arquivos em conflito
git diff - Ver detalhes dos conflitos
git merge --abort - Cancelar o merge e voltar ao estado anterior
git add [arquivo] - Marcar conflito como resolvido
git commit - Finalizar o merge após resolver conflitos

Dicas para Evitar Conflitos:

  • Faça pull frequentemente para manter seu código atualizado
  • Trabalhe em branches separadas para funcionalidades diferentes
  • Comunique-se com a equipe sobre quais arquivos estão sendo editados
  • Faça commits pequenos e frequentes
  • Evite reformatar arquivos inteiros desnecessariamente

Exemplo Prático de Fluxo de Trabalho

Cenário: Adicionando uma nova funcionalidade

1. Atualizar a branch principal:

git checkout main
git pull origin main

2. Criar branch para nova funcionalidade:

git checkout -b nova-funcionalidade

3. Trabalhar na funcionalidade:

# Editar arquivos...
git add .
git commit -m "Adiciona nova funcionalidade"

4. Enviar branch para GitHub:

git push origin nova-funcionalidade

5. Fazer merge na branch principal:

git checkout main
git merge nova-funcionalidade
git push origin main

6. Limpar (opcional):

git branch -d nova-funcionalidade

Boas Práticas

Mensagens de Commit:

  • Use mensagens claras e descritivas
  • Comece com um verbo no imperativo (Ex: "Adiciona", "Corrige", "Remove")
  • Seja específico sobre o que foi alterado
  • Mantenha a primeira linha com até 50 caracteres

Organização:

  • Uma branch para cada funcionalidade ou correção
  • Nomes de branches descritivos (ex: "corrige-bug-login")
  • Commits pequenos e focados em uma mudança específica
  • Sempre teste antes de fazer push

Lembre-se:

Git e GitHub podem parecer complicados no início, mas com a prática se tornam ferramentas indispensáveis. Comece devagar, pratique os comandos básicos e aos poucos você se sentirá confortável com fluxos mais complexos.

A chave é praticar! Crie um repositório de teste no GitHub e experimente os comandos. Não tenha medo de cometer erros - é assim que se aprende!

Próximos Passos

Agora que você conhece os conceitos básicos, pratique criando um repositório no GitHub, clonando-o localmente, fazendo algumas modificações, commitando e enviando de volta. Com o tempo, explore funcionalidades mais avançadas como rebasing, stashing e hooks.

Lembre-se: a melhor forma de aprender Git e GitHub é usando-os em projetos reais. Cada desenvolvedor tem seu próprio fluxo de trabalho, e você desenvolverá o seu conforme ganha experiência.
